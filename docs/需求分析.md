# 视频解析下载需求分析

## 1. 文档目的

本文用于沉淀当前阶段的视频解析下载需求、边界、平台限制和后续扩展方向，便于后续人类或 AI 在继续开发时快速理解项目真实目标，避免反复讨论已确认的取舍。

本文件应与以下文档一起阅读：

- `docs/PRD.md`：产品愿景和长期功能范围。
- `docs/方案设计.md`：当前工程方案、接口和扩展设计。
- `DESIGN/DESIGN.md`、`DESIGN/首页.html`：首页视觉和交互参考。

## 2. 当前阶段目标

本期只完成“视频解析 + 浏览器直接下载”的 Web MVP，不做 AI 总结、登录、会员、数据库、多用户队列或公网安全能力。

核心用户流程：

1. 用户在首页粘贴公开视频链接。
2. 前端调用后端解析接口，展示标题、作者、时长、封面和可用下载档位。
3. 用户选择 `4k`、`1080p`、`720p` 或 `audio`。
4. 浏览器直接接收后端返回的附件文件并保存。

当前定位是本地自用或内网验证版本，不是可直接开放给公网用户的大规模下载服务。

## 3. 用户需求

### 3.1 必须满足

- 支持用户粘贴公开视频链接并解析基础信息。
- 支持按统一档位下载视频或音频：
  - `4k`：4K 原画。
  - `1080p`：1080P 高清。
  - `720p`：720P 标清。
  - `audio`：纯音频 MP3。
- 不写死平台列表，平台能力以当前 `yt-dlp` 版本支持范围为准。
- 缺失档位不能误标为可下载。
- 文件大小拿不到时允许显示未知。
- 解析失败、下载失败、缺少 `ffmpeg`、平台限制等情况要给出中文提示。
- 前端首页视觉贴近蓝白 SaaS 风格设计稿，但本期主流程只保留下载工具。

### 3.2 本期不做

- AI 总结、字幕提取、思维导图、问答。
- 用户系统、登录、会员、支付、权益校验。
- 数据库、历史记录、收藏、下载任务持久化。
- 多用户下载队列、任务重试中心、后台任务管理。
- 公网部署安全策略。
- 让普通用户导出并输入 Cookie。
- 读取用户浏览器 Cookie 或处理登录/会员内容。

## 4. 竞品与平台限制分析

### 4.1 竞品观察

以 BibiGPT 为参考，公开资料显示其核心是音视频总结，而不是单纯“服务端万能下载器”。它同时提供：

- 多平台链接处理。
- 浏览器插件，辅助识别用户当前打开的视频页面。
- 本地音视频上传。
- 通过可访问的音视频文件 URL 触发处理。

因此，竞品更像是“服务端解析 + 浏览器插件 + 文件上传/文件 URL + 缓存”的混合路线，而不是要求用户手动复制 Cookie。

### 4.2 公开视频与受限视频的区别

服务器能否直接解析视频，取决于平台是否允许当前服务器请求直接访问视频元信息和媒体流。

常见失败原因：

- 视频需要登录、会员、年龄校验或地区权限。
- 平台对服务器 IP、请求频率或数据中心网络做了风控。
- 视频播放接口要求 Cookie、签名、浏览器指纹、Referer、User-Agent 等上下文。
- 高画质档位比低画质档位有更严格的授权要求。
- 平台接口变化导致 `yt-dlp` 暂时失效。

以 B 站为例，部分链接在浏览器中能打开，但服务器通过 `yt-dlp` 请求播放信息时可能返回 `HTTP Error 412: Precondition Failed`。这类问题通常不是简单换前端交互能解决，而是平台风控或登录态限制。

### 4.3 抖音专用 Provider

当前项目可为抖音增加实验性专用解析链路：抖音链接优先走 `DouyinProvider`，其他平台继续走 `yt-dlp`。

该能力的目标是提高抖音公开视频的解析成功率，但不承诺稳定绕过平台风控：

- 支持从抖音分享文案中提取第一个视频链接。
- 支持跟随短链重定向并提取视频 ID。
- 尝试调用 `iesdouyin` 视频信息接口；失败时尝试分享页 `_ROUTER_DATA` 兜底。
- 只把 `playwm` 到 `play` 作为候选无水印地址，不承诺一定无水印。
- 如果接口返回加密参数缺失、风控页面、地区/IP 限制或媒体地址缺失，应返回明确中文错误。

### 4.4 Cookie 策略

后续部署到服务器时，不建议让普通用户直接获取并输入 Cookie：

- 操作门槛高，用户体验差。
- Cookie 属于敏感凭证，泄露风险高。
- 服务端保存用户 Cookie 会带来合规、安全和责任问题。
- 平台可能禁止凭证共享或自动化批量下载。

更合理的策略：

- 默认只处理公开可访问链接。
- 对受限平台提示“当前平台限制服务器解析”。
- 如需自用增强，可由管理员在服务器侧配置受控凭证，但不暴露给普通用户。
- 面向正式用户时，优先考虑浏览器插件、桌面助手、本地上传或文件 URL 兜底。

## 5. 部署前必须补齐的需求

如果后续要部署到公网服务器，应先补齐以下产品和工程需求：

- 用户认证与配额。
- 限流和滥用防护。
- SSRF 防护，避免用户传入内网地址或恶意 URL。
- 下载任务队列，避免长请求阻塞 Web 进程。
- 文件大小、视频时长、任务耗时限制。
- 临时文件生命周期清理。
- 错误码分层和可观测日志。
- 平台可用性状态提示。
- 服务端资源隔离，避免单个任务占满 CPU、磁盘或带宽。

## 6. 后续功能扩展方向

### 6.1 AI 总结

未来 AI 总结可以复用当前解析结果，但不应依赖“必须下载完整视频”这一前提。更合理的输入来源包括：

- `yt-dlp` 提取字幕或自动字幕。
- 直接从音频流抽取音频并转写。
- 用户上传本地音视频文件。
- 用户提供可访问的文件 URL。

### 6.2 浏览器插件

浏览器插件适合解决以下问题：

- 用户无需手动处理 Cookie。
- 插件运行在用户自己的浏览器上下文中，可以识别当前页面。
- 对服务器无法解析的平台，插件可作为辅助入口。

插件不应默认把用户 Cookie 发送到服务器；如涉及页面内容、媒体地址或文件上传，需要单独设计权限、告知和安全边界。

### 6.3 任务系统

当功能从“直接下载”扩展到“下载、转码、转写、总结、导出”时，应引入异步任务模型：

- 创建任务。
- 查询任务状态。
- 获取任务结果。
- 失败重试和错误归因。
- 临时文件或产物的过期清理。

## 7. 验收标准

当前 MVP 的验收标准：

- 能启动前后端服务。
- 健康检查接口返回后端可用状态和 `ffmpeg` 状态。
- 输入有效公开视频链接后，能返回视频基础信息。
- 前端能展示解析结果和可用档位。
- 点击可用档位后，浏览器发起附件下载。
- 无效 URL 或平台限制时，前端展示中文错误提示。
- 前端构建通过。
- 后端最小测试通过。

对 B 站等存在风控的平台，验收口径不是“所有公开视频都必须可下载”，而是“服务端能正确识别失败并给出明确提示”。
