# 视频解析下载方案设计

## 1. 当前架构

当前项目采用轻量前后端分离架构：

- 后端：`FastAPI + yt-dlp + httpx + ffmpeg + StepAudio 2.5 ASR + DeepSeek`，其中 B 站公开视频优先走内置网页解析 Provider。
- 前端：`React + Vite + Tailwind CSS + lucide-react`。
- 存储：无数据库，下载文件落在临时目录，请求结束后清理。
- 部署目标：当前优先本地自用，后续如公网部署需补安全、持久化任务系统、配额和资源隔离。
- 配置：后端启动时读取项目根目录 `.env`，用于 `STEP_API_KEY`、StepAudio ASR 地址、STT 限制参数、`DEEPSEEK_API_KEY`、DeepSeek 模型和总结会话 TTL。

目录结构：

```text
backend/
  app/
    main.py            # FastAPI 入口和接口定义
    models.py          # Pydantic 请求/响应模型
    video_service.py   # Provider 调度、档位判断、下载入口
    deepseek_client.py # DeepSeek Chat Completions 封装
    summary_service.py # AI 总结编排、SSE 事件和问答会话
    summary_models.py  # 摘要、思维导图、问答请求模型
    prompt_templates.py# 总结、脑图和问答提示词
    providers/         # 平台 Provider，B 站和抖音优先走专用链路
  tests/
    test_video_service.py
frontend/
  src/
    App.tsx            # 首页和下载主流程
    App.css            # 页面样式，已接入 Tailwind CSS
    api.ts             # 前端 API 封装
    types.ts           # 前端类型
docs/
  PRD.md
  需求分析.md
  方案设计.md
  B站解析接入记录.md
```

## 2. 关键设计决策

### 2.1 yt-dlp 与平台专用 Provider

本期不 fork 或深度修改 `yt-dlp`，而是在后端直接封装其 Python API，并针对少数平台补充专用 Provider。

原因：

- `yt-dlp` 已覆盖大量公开视频平台，维护成本低于自研解析器。
- 站点解析规则变化频繁，直接复用上游更新更稳妥。
- MVP 阶段只需要元信息解析和下载，不需要修改其内部 extractor。

当前使用方式：

- `extract_info(download=False)` 获取元信息和 formats。
- `ydl.download([url])` 下载指定格式。
- 通过 `ffmpeg` 合并音视频或导出 MP3。

B 站公开视频优先走 `BilibiliProvider`：

- 请求视频页并读取页面公开脚本中的 `window.__playinfo__` 和 `window.__INITIAL_STATE__`。
- 从 DASH 数据中选择视频流和音频流。
- 页面缺少可用 `__playinfo__` 时，用 `bvid + cid` 请求 B 站公开播放地址接口兜底。
- 同时合并普通 MP4 `durl` 流，避免匿名 DASH 只返回低清时误把低清当成最高可用视频。
- 通过 `x/player/v2` 获取匿名可访问字幕列表，并下载字幕 JSON 转成结构化文本；字幕不可用不影响视频解析成功。
- 下载媒体流时携带原页面 `Referer` 和浏览器 User-Agent。
- 下载阶段使用 `ffmpeg` 合并视频和音频，音频档位导出为 MP3。
- 登录、会员、高画质权限、风控、地区/IP 限制仍不承诺绕过。

更完整的接入取舍、第三方解析 API 对比、清晰度限制和排障记录见 `docs/B站解析接入记录.md`。

### 2.2 不引入数据库

本期接口是同步请求模型：

- 解析接口立即返回元信息。
- 下载接口完成下载后直接以附件返回。
- 下载临时目录由 `BackgroundTask` 清理。
- 无公开字幕或字幕解析失败时，解析接口会自动创建内存转写任务；前端通过任务 ID 轮询状态和结果。

这样实现简单，但不适合公网高并发和大文件下载。当前 AI 总结仍保持本地内存态：总结流式接口一次性串联解析、字幕 / STT、摘要、思维导图和问答会话准备；问答上下文只保存在后端内存并按 TTL 清理。后续如加入多用户能力、历史记录或公网部署，应改为持久化异步任务模型。

### 2.3 AI 视频总结

AI 总结能力通过 `POST /api/summaries/stream` 暴露，使用 `fetch` 读取 SSE，事件包括阶段进度、视频信息、文稿、摘要增量、摘要完成、思维导图完成、问答准备完成、部分失败和致命失败。

设计边界：

- DeepSeek API Key 只读取后端 `.env`，不返回给前端。
- 默认模型为 `deepseek-v4-flash`，普通摘要、思维导图和问答默认关闭 thinking mode；可通过 `DEEPSEEK_THINKING_ENABLED=true` 开启。
- 摘要使用流式 Markdown 输出，前端直接形成打字机效果；后端再用确定性规则生成辅助结构化摘要。
- 思维导图使用独立 JSON Output 调用，并通过 `MindMapNode` 校验后再交给前端。
- 问答会话只保存在内存中，默认 TTL 为 24 小时，页面刷新不提供恢复入口。
- 思维导图或问答准备失败时，摘要和原文稿仍可继续展示。
- 自定义提示词只作为摘要正文的附加关注点，不允许覆盖系统边界、输出结构或后续问答规则。

### 2.4 不让用户输入 Cookie

当前不支持前端输入 Cookie，也不读取用户浏览器 Cookie。

原因：

- Cookie 是敏感凭证。
- 用户手动导出和粘贴 Cookie 体验差。
- 公网服务保存用户 Cookie 风险高。
- 平台风控可能还依赖 IP、浏览器指纹和请求上下文，Cookie 不是万能解法。

后续如果需要增强受限平台支持，应优先考虑：

- 管理员服务器侧受控凭证，仅用于自用或明确授权场景。
- 浏览器插件或桌面助手。
- 用户上传本地文件或提供可访问文件 URL。

### 2.5 抖音专用 Provider

抖音链接优先走实验性 `DouyinProvider`，其他平台继续走 `YtDlpProvider`。

设计边界：

- 支持 `douyin.com`、`v.douyin.com`、`iesdouyin.com`。
- 支持从分享文本中提取第一个 `http/https` 链接。
- 使用 `httpx` 处理重定向、超时和流式下载。
- 解析链路优先尝试 `iesdouyin` item info 接口，失败时尝试移动端分享页 `_ROUTER_DATA`。
- `playwm` 到 `play` 的替换只作为候选地址，不承诺稳定无水印。
- 当前抖音只暴露 `source` 原视频和 `audio` 音频档位，`4k/1080p/720p` 默认不可用。
- 风控、加密参数缺失、接口变化、地区/IP 限制时返回明确中文错误。

## 3. 后端设计

### 3.1 接口

#### `GET /api/health`

返回服务状态和 `ffmpeg` 可用性。

```json
{
  "status": "ok",
  "ffmpegAvailable": true,
  "sttAvailable": true,
  "deepseekAvailable": true
}
```

#### `POST /api/videos/parse`

请求：

```json
{
  "url": "https://example.com/video"
}
```

响应：

```json
{
  "title": "视频标题",
  "uploader": "作者",
  "duration": 1234,
  "thumbnail": "https://example.com/cover.jpg",
  "webpageUrl": "https://example.com/video",
  "subtitleStatus": "available",
  "subtitleMessage": null,
  "subtitles": [
    {
      "language": "zh-CN",
      "languageLabel": "中文",
      "text": "第一句字幕\n第二句字幕",
      "cues": [
        {
          "start": 0.0,
          "end": 1.5,
          "text": "第一句字幕"
        }
      ]
    }
  ],
  "transcriptTask": {
    "taskId": "9f7c...",
    "status": "queued",
    "source": "asr",
    "message": "等待开始自动转写。",
    "text": null
  },
  "options": [
    {
      "quality": "1080p",
      "label": "1080P 高清",
      "available": true,
      "estimatedSize": 123456789
    }
  ]
}
```

当 `subtitleStatus=available` 且存在公开字幕时，`transcriptTask` 为 `null`，前端直接展示字幕文稿。当公开字幕不可用、需要登录、字幕接口失败或字幕文件格式异常时，后端自动创建 STT 任务并返回 `transcriptTask`。

#### `GET /api/videos/download`

查询参数：

- `url`：公开视频链接。
- `quality`：`4k`、`1080p`、`720p`、`audio`。

成功时返回 `Content-Disposition: attachment` 的文件响应。

失败时返回 FastAPI 标准 JSON 错误：

```json
{
  "detail": "中文错误说明"
}
```

#### `POST /api/transcripts`

请求：

```json
{
  "url": "https://example.com/video"
}
```

响应为转写任务状态，适合手动补创建任务：

```json
{
  "taskId": "9f7c...",
  "status": "queued",
  "source": "asr",
  "message": "等待开始自动转写。",
  "text": null
}
```

#### `GET /api/transcripts/{taskId}`

返回当前转写任务状态和文稿结果：

```json
{
  "taskId": "9f7c...",
  "status": "completed",
  "source": "asr",
  "message": "文稿生成完成。",
  "text": "完整视频文稿..."
}
```

当前任务状态保存在内存中，服务重启或任务过期后会返回“转写任务不存在或已过期”。

#### `POST /api/summaries/stream`

请求：

```json
{
  "url": "https://example.com/video",
  "style": "study_notes",
  "customPrompt": "请重点提炼产品策略和可执行建议"
}
```

响应为 `text/event-stream`，前端通过 `fetch` 读取事件流。核心事件：

- `stage`：阶段进度。
- `video`：视频基础信息和下载档位。
- `transcript`：公开字幕或 StepAudio ASR 文稿。
- `summary_delta`：摘要 Markdown 增量。
- `summary_done`：完整 Markdown 和辅助结构化摘要。
- `mindmap_done`：通过后端校验的树形思维导图。
- `qa_ready`：临时问答会话 ID。
- `partial_error`：思维导图或问答等部分能力失败。
- `fatal_error`：无法继续总结的错误。

#### `POST /api/summaries/{sessionId}/questions/stream`

请求：

```json
{
  "question": "视频里提到的三个关键方法分别是什么？"
}
```

响应为 `text/event-stream`，事件包括：

- `answer_delta`：回答增量。
- `answer_done`：当前回答完成。
- `fatal_error`：会话过期、DeepSeek 调用失败等错误。

### 3.2 档位选择

后端统一暴露五个产品档位：

| 档位 | 展示名称 | 选择思路 |
| --- | --- | --- |
| `source` | 原视频 MP4 | 抖音等专用 Provider 返回的原始媒体地址 |
| `4k` | 4K 原画 | 选择 `height>=2160` 的最佳视频并合并最佳音频 |
| `1080p` | 1080P 高清 | 选择 `height=1080` 的最佳视频并合并最佳音频 |
| `720p` | 720P 标清 | 选择 `height=720` 的最佳视频并合并最佳音频 |
| `audio` | 纯音频 MP3 | 选择最佳音频并通过 ffmpeg 导出 MP3 |

解析阶段只根据平台返回的公开视频信息判断档位是否可用，不保证下载阶段一定成功。下载阶段仍可能因平台风控、临时链接失效、网络或 `ffmpeg` 错误失败。

### 3.3 错误处理

后端将常见 Provider 错误转换为中文提示：

- 不支持链接。
- 需要登录、会员或 Cookie。
- 平台返回 412 访问限制。
- 视频不可用。
- 网络访问失败。
- `ffmpeg` 缺失或处理失败。

后续建议将字符串错误升级为结构化错误码：

```json
{
  "code": "RISK_CONTROL",
  "message": "视频平台限制服务器解析",
  "retryable": false
}
```

建议错误码：

- `INVALID_URL`
- `UNSUPPORTED_URL`
- `AUTH_REQUIRED`
- `RISK_CONTROL`
- `VIDEO_UNAVAILABLE`
- `FFMPEG_MISSING`
- `NETWORK_ERROR`
- `DOWNLOAD_FAILED`

## 4. 前端设计

### 4.1 页面结构

首页和结果页延续蓝白 SaaS 风格，当前保留以下核心区块：

- 顶部导航和品牌。
- AI 视频总结主入口。
- 链接输入框、风格选择和自定义总结提示词。
- 阶段式进度条。
- 视频封面预览和原站打开按钮。
- 智能总结、思维导图、原文稿、问答 Tabs。
- 下载档位作为结果页附加操作。

不再把会员、价格或营销区块作为当前 MVP 主流程的一部分。

### 4.2 状态流

前端核心状态：

- `url`：用户输入链接。
- `style`：总结风格。
- `customPrompt`：用户自定义总结关注点。
- `isRunning`：总结生成中。
- `stages`：阶段式进度。
- `video`：视频基础信息和下载档位。
- `transcript`：公开字幕或 StepAudio ASR 文稿。
- `summaryMarkdown`：流式摘要 Markdown。
- `mindmap`：后端校验后的树形思维导图。
- `qaSessionId`：后端内存问答会话 ID。
- `qaMessages`：当前页面内临时问答历史。
- `error`：错误信息。
- `notice`：状态提示。
- `activeQuality`：当前触发下载的档位。

交互流程：

1. 用户提交表单。
2. 前端先做 `http/https` URL 校验。
3. 调用 `POST /api/summaries/stream`。
4. 前端读取 SSE，逐步更新进度、视频信息、文稿和摘要增量。
5. 摘要完成后展示思维导图、原文稿和问答入口。
6. 点击下载档位后复用现有流式保存逻辑，保留页内下载进度。

### 4.3 Tailwind CSS

项目已接入 Tailwind CSS v4：

- `frontend/package.json` 包含 `tailwindcss` 和 `@tailwindcss/vite`。
- `frontend/vite.config.ts` 使用 Tailwind Vite 插件。
- `frontend/src/App.css` 引入 `@import "tailwindcss";`。

当前样式仍主要保留在 `App.css` 中，避免为了引入 Tailwind 而大规模改写稳定 UI。后续新增组件可逐步使用 Tailwind utility class。

## 5. 本地开发与验证

### 5.0 配置

自动 STT 和 AI 总结需要在项目根目录创建 `.env`：

```bash
cp .env.example .env
```

最小配置：

```env
STEP_API_KEY=你的 StepFun API Key
DEEPSEEK_API_KEY=你的 DeepSeek API Key
```

`.env` 已加入 `.gitignore`，不要提交真实密钥。后端会优先读取 `.env`，未配置时 `sttAvailable=false` 或 `deepseekAvailable=false`；视频解析和下载仍可使用，但对应 STT 或总结能力会返回清晰错误。

STT 限制参数使用便于填写的单位：

- `STEP_ASR_MAX_DURATION_MINUTES`：单个视频自动转写最长时长，单位分钟。
- `STEP_ASR_MAX_AUDIO_FILE_MB`：单个转写音频文件大小上限，单位 MB。
- `DEEPSEEK_MODEL`：AI 总结模型，默认 `deepseek-v4-flash`。
- `DEEPSEEK_THINKING_ENABLED`：是否开启 DeepSeek 思考模式，默认 `false`。开启后后端只消费最终 `content`，不向前端展示 `reasoning_content`。
- `AI_SUMMARY_SESSION_TTL_SECONDS`：总结问答内存会话保留时间，默认 24 小时。

## 6. 公网部署演进方案

当前同步下载模型不建议直接公网开放。推荐演进为：

### 6.1 第一阶段：内网或自用部署

- 保持当前 FastAPI + React 架构。
- 增加基本配置项：
  - 最大 URL 长度。
  - 最大下载时长。
  - 最大文件大小。
  - 临时目录路径。
  - 请求超时。
- 增加反向代理和基础访问控制。

### 6.2 第二阶段：公网 MVP

- 引入用户登录。
- 引入限流和配额。
- 引入 SSRF 防护。
- 下载改为异步任务：
  - `POST /api/tasks` 创建任务。
  - `GET /api/tasks/{id}` 查询状态。
  - `GET /api/tasks/{id}/file` 下载结果。
- 临时文件按任务 ID 存放并设置 TTL。
- 后台 worker 负责下载、转码和清理。

### 6.3 第三阶段：AI 总结持久化

- 把当前内存态总结任务迁移为持久化任务。
- 为总结、文稿、思维导图和问答历史增加用户归属和 TTL。
- 增加配额、成本估算和后台 worker。
- 前端新增历史列表或任务详情页。

### 6.4 第四阶段：浏览器插件

- 插件识别当前视频页面。
- 插件将页面 URL、标题、封面、必要的页面上下文发送给后端。
- 对服务端无法解析的视频，插件提供本地上传或本地处理入口。
- 插件不得默认上传 Cookie；任何敏感数据处理都需要单独设计权限和用户告知。

## 7. 安全注意事项

公网部署前必须处理：

- URL 协议白名单，仅允许 `http` 和 `https`。
- 禁止访问内网、localhost、云元数据地址等敏感网段。
- 限制重定向次数和最终目标地址。
- 限制文件大小、下载速度、任务时长。
- 限制并发数和单用户配额。
- 不在日志中打印 Cookie、Authorization、完整敏感 URL。
- 临时文件必须定时清理。
- 后端下载进程需要资源隔离，避免耗尽磁盘、CPU、内存或带宽。

## 8. 给后续 AI 的开发提示

后续继续开发时，应遵守以下约束：

- 先阅读 `AGENTS.md`、`README.md`、`docs/PRD.md`、`docs/需求分析.md` 和本文档。
- 当前目标仍以下载 MVP 为基线，不要擅自加入 AI、数据库、登录或支付。
- 如果用户要求公网部署，先补安全、任务系统和限制策略，不要直接暴露当前下载接口。
- 不要添加用户 Cookie 输入框，除非用户明确改变产品策略。
- 站点解析优先复用 `yt-dlp`，不要轻易自研平台解析器。
- 遇到 B 站 412 或类似平台风控，不要承诺通过代码必然绕过，应返回清晰错误并提供插件、上传或稍后重试路径。
- 修改前端视觉时继续贴近 `DESIGN/首页.html`，保持蓝白 SaaS 风格。
- 新增依赖前先确认必要性，并记录原因。
