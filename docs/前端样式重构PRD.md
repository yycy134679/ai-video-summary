# PRD: 模仿首页设计稿重构前端样式

## 1. 问题陈述

当前项目已经具备公开视频解析、字幕 / ASR 文稿、AI 摘要、思维导图、临时问答和下载进度等核心能力，但前端视觉还需要进一步贴近 `DESIGN/首页.png` 所呈现的完整 SaaS 官网质感。用户首次打开页面时，应能在首屏快速理解“粘贴视频链接即可节省大量观看时间”，并通过高质量的产品预览、功能说明和价格卡片建立对工具专业度的信任。

本次重构聚焦前端样式、组件结构和静态展示信息，不改变后端接口、总结流程、下载流程或当前本地自用 MVP 的产品边界。

## 2. 背景与依据

- 当前项目定位为本地自用 MVP，无账号、会员、数据库、多用户队列或公网部署安全策略。
- 核心流程是粘贴公开视频链接，优先读取公开字幕；无公开字幕或字幕解析失败时使用 StepAudio 2.5 ASR 生成文稿，再调用 DeepSeek 生成结构化摘要、思维导图和临时问答会话。
- 现有前端技术栈为 `React + Vite + TypeScript + Tailwind CSS v4 + lucide-react`，入口在 `frontend/src/App.tsx`，主要样式在 `frontend/src/App.css`。
- `DESIGN/首页.png` 展示的是蓝白 SaaS 官网风格，包括顶部导航、居中 Hero、视频链接输入框、能力标签、产品预览、信任背书、功能卡片和三列价格卡片。
- `DESIGN/DESIGN.md` 明确视觉方向为现代专业主义，强调信任蓝、白色卡片、柔和阴影、通透感、秩序感和克制圆角。

## 3. 目标

1. 首页视觉高度贴近 `DESIGN/首页.png`，包含蓝白渐变背景、顶部导航、升级提示胶囊、主标题、链接输入框、主 CTA、能力标签、产品预览、功能卡片和价格卡片。
2. 保留现有核心业务流程：输入公开视频链接、选择总结风格、生成摘要、展示阶段进度、摘要、脑图、原文稿、问答和下载状态。
3. 将基础 UI 质量提升到生产级水准，允许以打造更好 UI 设计为目标替换所有基础按钮、输入框、卡片和 Tab。
4. 允许引入 `shadcn/ui`，用于提升 Button、Card、Input、Textarea、Tabs、Badge、Alert 等基础组件的可访问性、一致性和可维护性。
5. 允许组件拆分，降低 `App.tsx` 复杂度，使首页、总结页和基础 UI 组件有更清晰的边界。
6. 桌面端尽量还原设计稿观感，移动端保证无横向溢出且主流程可完成。

## 4. 非目标

1. 不实现真实登录、注册、免费试用、订阅、支付、会员权益校验或团队协作。
2. 不新增后端接口、数据库、历史记录、多用户任务队列、持久化任务系统或公网部署安全能力。
3. 不改变视频解析、字幕优先、STT 兜底、DeepSeek 总结、问答会话和下载逻辑。
4. 不接入真实客户 Logo、真实商业案例或真实价格权益系统。
5. 不为了引入 `shadcn/ui` 做无关的大范围业务重构。

## 5. 设计原则

### 5.1 视觉方向

本次视觉方向为精致、可信、通透的蓝白 SaaS 官网。应遵循 `frontend-design` 的高质量前端要求，但在本项目中不追求夸张视觉实验，而是围绕 `DESIGN/首页.png` 做克制、完整、专业的落地。

关键视觉特征：

- 大面积白色和浅蓝背景。
- 主蓝色用于 CTA、重点数字、激活态和可交互状态。
- 柔和蓝色阴影与玻璃拟态卡片。
- 居中 Hero 和强层级标题。
- 大圆角输入框、卡片和产品预览容器。
- 轻量弧形光带、渐变光晕或背景装饰，用于模拟设计稿中的空间纵深。

### 5.2 信息边界

设计稿中的导航、登录、免费试用和价格卡片需要作为静态视觉内容保留，但不得暗示当前已经具备真实账号、支付或权益系统。按钮文案必须完全照设计稿，不额外加“静态展示”等说明；但点击行为应避免进入不存在的业务流程。

### 5.3 组件策略

- 可以引入 `shadcn/ui`。
- 可以替换所有基础按钮、输入框、卡片和 Tab。
- 应优先复用少量稳定基础组件，而不是堆叠过多 UI 组件。
- `shadcn/ui` 按其“组件复制到项目内维护”的模式使用，避免把它当作黑盒组件库。
- 组件拆分应服务可读性和样式复用，不做与当前任务无关的架构重写。

## 6. 用户故事

### 6.1 首次访问用户

- 作为首次访问用户，我希望首页首屏清楚展示“AI 视频总结，节省 80% 时间”，以便立即理解产品价值。
- 作为首次访问用户，我希望看到醒目的输入框和“开始总结”按钮，以便知道下一步该做什么。
- 作为首次访问用户，我希望通过产品预览理解最终会得到摘要、章节和关键片段，而不是只看到抽象营销文案。

### 6.2 学习型用户

- 作为学习型用户，我希望快速粘贴课程、演讲或访谈链接，以便生成可复习的结构化笔记。
- 作为学习型用户，我希望结果页保留摘要、脑图、原文稿和问答入口，以便从不同角度回看视频内容。
- 作为学习型用户，我希望处理中状态清楚展示当前阶段，以便知道系统正在解析、转写或生成总结。

### 6.3 知识工作者

- 作为知识工作者，我希望页面看起来专业可信，以便愿意把长视频内容沉淀成工作资料。
- 作为知识工作者，我希望复制、导出、下载和追问入口保持清晰，以便把结果用于后续整理。

### 6.4 本地 MVP 使用者

- 作为本地部署用户，我希望导航和价格卡片视觉完整，以便体验接近正式产品官网。
- 作为本地部署用户，我希望静态导航和价格按钮不会触发未实现功能错误，以免干扰核心总结流程。

## 7. 功能需求

### 7.1 P0 必须实现

#### P0-1 首页 Hero 区严格参考设计稿

需求：

- 顶部导航包含品牌 Logo / 名称。
- 导航项保留：“功能”、“使用场景”、“定价”、“博客”、“帮助中心”。
- 右侧保留“登录”和“免费试用”。
- 导航项作为静态锚点或静态入口保留，不跳转到不存在页面。
- Hero 区包含升级提示胶囊、主标题、副标题、链接输入框、主 CTA、支持平台说明和能力亮点。
- 主标题重点突出“80%”。
- 链接输入框是实际可用的总结入口，继续触发现有 `streamVideoSummary` 流程。

验收标准：

- Given 用户打开首页，When 页面加载完成，Then 首屏信息结构应明显对应 `DESIGN/首页.png`。
- Given 用户输入合法公开视频链接，When 点击 Hero 中的开始按钮，Then 进入现有总结流程。
- Given 用户输入空链接或非法 URL，When 点击开始按钮，Then 显示现有中文错误提示。

#### P0-2 产品预览区完全模拟设计稿

需求：

- 产品预览区使用模拟数据，不依赖真实解析结果。
- 预览结构模拟设计稿中的视频卡片、AI 总结、章节要点和关键片段。
- 模拟内容需贴合项目真实能力，包括 AI 总结、章节概览、关键片段、文稿 / 字幕等。
- 预览区不得触发真实视频播放或真实下载。

验收标准：

- Given 用户未输入链接，When 查看首页，Then 仍能看到完整产品预览。
- Given 用户点击预览区，Then 不应误触发真实总结、下载或播放流程。
- Given 页面在桌面端展示，Then 预览区视觉层级应接近设计稿的大白色悬浮卡片。

#### P0-3 严格保留价格卡片

需求：

- 保留设计稿中的三列价格卡片结构：免费版、专业版、团队版。
- 价格卡片按钮文案完全照设计稿。
- 免费版按钮文案为“立即开始”。
- 专业版按钮文案为“立即订阅”。
- 团队版按钮文案为“联系销售”。
- 专业版保留突出态和“最受欢迎”标识。
- 按钮为静态展示或锚点行为，不接入支付、账号或销售流程。

验收标准：

- Given 用户滚动到定价区，When 查看价格卡片，Then 能看到免费版、专业版、团队版三列布局。
- Given 用户查看按钮，Then 按钮文案必须分别为“立即开始”、“立即订阅”、“联系销售”。
- Given 用户点击价格按钮，Then 不应进入不存在的支付、登录或销售页面。

#### P0-4 替换和统一基础 UI 组件

需求：

- 允许替换所有基础按钮、输入框、卡片和 Tab。
- 优先使用 `shadcn/ui` 或基于同等质量实现的本地基础组件。
- 替换后必须保留现有交互逻辑、禁用态、加载态、错误态和可访问性属性。
- 所有可交互组件应有清晰 hover、focus、disabled 状态。

验收标准：

- Given 用户使用键盘导航，When focus 到输入框、按钮或 Tab，Then 能看到清晰 focus 状态。
- Given 总结正在运行，When 主按钮处于加载态，Then 用户能明确知道正在处理且不能重复提交。
- Given 替换 Tab 组件，When 用户切换摘要、脑图、原文稿和问答，Then 内容切换逻辑保持不变。

#### P0-5 总结结果页视觉统一

需求：

- 结果页应与新首页共享颜色、圆角、阴影、卡片和按钮语言。
- 保留阶段进度、视频信息、摘要、脑图、原文稿、问答、下载、复制和导出能力。
- 部分失败和致命错误提示仍需清晰可见。
- 下载进度状态不得被隐藏或弱化。

验收标准：

- Given 总结流正在返回，When 摘要增量出现，Then 用户能在统一卡片容器中看到流式内容。
- Given 思维导图或问答准备失败，When 摘要已生成，Then 页面仍保留摘要并显示部分失败提示。
- Given 用户触发下载，When 下载进行中，Then 下载进度保持可见。

#### P0-6 响应式适配

需求：

- 桌面端重点还原 `DESIGN/首页.png`。
- 平板端保持合理卡片栅格和阅读层级。
- 移动端单列展示，输入框、CTA、Tab、问答输入和价格卡片不得横向溢出。

验收标准：

- Given 视口宽度为 1440px，Then 首页应接近设计稿的桌面布局。
- Given 视口宽度为 768px，Then 首页和结果页无横向滚动。
- Given 视口宽度为 390px，Then 用户可以完成输入、提交、查看结果和问答。

#### P0-7 构建验证

需求：

- 完成前端改动后必须运行 `cd frontend && npm run build`。

验收标准：

- TypeScript 检查通过。
- Vite 构建通过。

### 7.2 P1 优先增强

#### P1-1 背景与装饰层增强

需求：

- 添加类似设计稿的蓝色弧形光带、浅蓝渐变、玻璃拟态或柔和光晕。
- 装饰层不得遮挡核心内容或降低文本可读性。

#### P1-2 首页功能卡片区

需求：

- 保留“为什么选择 VideoSummarize AI?”结构。
- 功能卡片覆盖准确提炼重点、节省大量时间、多语言 / 多平台支持、安全和隐私保护等能力。
- 文案应贴合当前真实 MVP 能力，不夸大账号、团队、支付或云端历史能力。

#### P1-3 轻量动效

需求：

- 可以添加 Hero 区载入、卡片 hover、输入框 focus、按钮 hover、阶段状态变化等轻量动效。
- 动效应克制，不造成眩晕、不影响性能、不误导处理状态。

#### P1-4 组件目录整理

需求：

- 推荐拆分为 `frontend/src/components/home/*`、`frontend/src/components/summary/*`、`frontend/src/components/ui/*`。
- 拆分后 `App.tsx` 主要负责状态编排和页面切换。

### 7.3 P2 未来考虑

#### P2-1 真实官网能力

未来如果引入账号、付费、团队版或帮助中心，再将当前静态导航、登录、免费试用和价格按钮升级为真实入口。

#### P2-2 设计 Token 化

未来可将颜色、阴影、圆角、间距和排版抽离为更系统的 CSS variables 或 Tailwind theme 配置。

#### P2-3 前端测试与视觉回归

未来可补充组件测试、关键流程 E2E 或截图对比，避免后续样式回归。

## 8. shadcn/ui 使用约束

1. `shadcn/ui` 只作为提升基础组件质量的工具，不改变产品业务边界。
2. 引入前需确认与当前 Vite、React、TypeScript、Tailwind CSS v4 配置兼容。
3. 需要配置 `components.json`、路径别名和工具函数时，应适配当前项目目录结构。
4. 组件应复制到项目内维护，允许按本项目蓝白视觉系统二次定制。
5. 不应一次性引入大量暂未使用组件。
6. 新增依赖必须服务于 UI 质量提升，避免引入大型无关依赖。

## 9. 成功指标

### 9.1 领先指标

- 首页首屏 5 秒内能让测试者说出产品用途和核心操作入口。
- 桌面端首页与 `DESIGN/首页.png` 的布局和视觉层级高度一致。
- 移动端 390px 宽度无横向滚动。
- 主流程从输入链接到进入总结不超过一个主要 CTA。
- `cd frontend && npm run build` 通过。

### 9.2 滞后指标

- 本地试用反馈中“界面像 Demo / 不完整”的负面评价减少。
- 用户更愿意用该工具处理课程、访谈、发布会等长视频。
- 后续新增前端模块时能复用统一基础组件和视觉规范，减少样式冲突。
