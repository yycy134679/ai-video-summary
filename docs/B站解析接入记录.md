# B 站解析接入记录

## 1. 背景

本轮目标是评估并接入 `wwwzhouhui/video-parser` 对 B 站视频的解析思路，用它替换 B 站场景下对 `yt-dlp` 的依赖，同时保留项目当前边界：

- 只处理公开可访问的视频。
- 不读取浏览器 Cookie。
- 不让普通用户手动输入 Cookie。
- 不承诺绕过登录、会员、地区、风控或高画质权限限制。
- 其他平台仍继续走 `yt-dlp`。

## 2. video-parser 的实现方式

`video-parser` 对 B 站的关键思路不是破解 Cookie，也不是绕过会员权限，而是复用公开视频页面里已经暴露的播放数据：

1. 请求 B 站视频页 HTML。
2. 从页面脚本里读取 `window.__INITIAL_STATE__`，用于获取标题、作者、封面、`bvid`、`cid` 等基础信息。
3. 从页面脚本里读取 `window.__playinfo__`，用于获取公开播放流。
4. 如果页面里没有可用播放信息，再用 `bvid + cid` 请求 B 站公开播放地址接口。
5. DASH 流通常是视频和音频分离的，需要分别下载后用 `ffmpeg` 合并。
6. 媒体下载请求需要带上浏览器 `User-Agent` 和原视频页 `Referer`，否则部分媒体地址会被拒绝。

这个方案的能力上限取决于匿名访问时 B 站返回了哪些流。匿名状态能拿到低清或普通 MP4，不代表能拿到 1080P、4K、会员或登录后才有的流。

## 3. 与第三方解析 API 的取舍

对比 `https://www.xiazaitool.com/jiekou` 这类第三方视频解析 API，当前项目优先选择内置 Provider。

内置 Provider 的优势：

- 请求链路、错误处理、下载文件、临时目录清理都在本项目内可控。
- 不把用户输入的视频链接转交给未知第三方服务。
- 不依赖第三方 API 的稳定性、限流策略、收费策略或下线风险。
- 可以按项目需要精确控制“不读 Cookie、不绕过权限”的安全边界。
- 便于写测试和针对真实失败原因做中文提示。

内置 Provider 的代价：

- B 站页面和接口结构变化时，需要自己维护解析逻辑。
- 匿名访问拿不到的高画质，代码本身无法补出来。
- 需要处理 DASH 合并、Content-Type 校验、临时文件清理等工程细节。

第三方解析 API 的优势：

- 接入快，前期代码少。
- 有些 API 可能已经内置多个平台的临时兼容逻辑。

第三方解析 API 的风险：

- 黑盒，不清楚它是否使用 Cookie、代理池、盗链或其他高风险方式。
- 可用性、限额、费用、接口变更不可控。
- 用户链接会流向第三方，不适合默认作为本地自用 MVP 的核心依赖。
- 一旦 API 返回异常，很难定位是平台限制、API 故障还是本项目问题。

结论：本地自用 MVP 优先使用内置 B 站 Provider；第三方解析 API 只适合作为未来可选兜底，并且需要放在明确的 Provider 边界后面，默认关闭。

## 4. 当前接入方案

当前后端调度顺序：

1. 抖音链接优先走 `DouyinProvider`。
2. B 站链接优先走 `BilibiliProvider`。
3. 其他平台走 `YtDlpProvider`。

`BilibiliProvider` 的主要行为：

- 支持 `bilibili.com` 和 `b23.tv`。
- 从输入文本中提取第一个 `http/https` URL。
- 解析 `window.__INITIAL_STATE__` 和真正的 `window.__playinfo__ = {...}` 赋值。
- 页面缺少 `__playinfo__` 时，通过 `x/player/playurl` 用 `bvid + cid` 拉取公开播放地址。
- 同时请求 DASH 数据和普通 MP4 `durl` 数据，避免只拿到模糊的匿名 DASH 低清流。
- `source` 档位表示“最高可用 MP4”，不是平台意义上的所有权限最高画质。
- DASH 分离流通过 `ffmpeg` 合并为 MP4。
- 音频档位通过 `ffmpeg` 导出 MP3。
- 媒体下载带原页面 `Referer`、浏览器 `User-Agent` 和 `Origin`。

## 5. 清晰度限制

B 站的清晰度要区分“产品展示档位”和“匿名接口实际返回的流”。

当前统一展示：

| 档位 | 含义 |
| --- | --- |
| `source` | 当前匿名访问能拿到的最高可用 MP4 |
| `4k` | 需要返回 2160P 或以上视频流 |
| `1080p` | 需要返回 1080P 视频流 |
| `720p` | 需要返回 720P 视频流 |
| `audio` | 需要返回可下载音频流 |

对示例链接 `https://www.bilibili.com/video/BV1mAAmzqEfP/`，匿名状态下可以拿到 720P 普通 MP4，因此 `source` 和 `720p` 可用；1080P 和 4K 不可用。这不是前端选择问题，而是匿名播放接口没有返回对应权限的媒体流。

如果用户反馈“原视频很糊”，优先检查：

- 是否只使用了匿名 DASH 流，导致只拿到 480P 或 360P。
- 是否同时请求了普通 MP4 `durl` 数据。
- `source` 是否真的按可用候选里最高高度、最高码率排序。
- 前端展示文案是否避免误导用户以为 `source` 一定是原站最高画质。

## 6. 已处理的问题

### 6.1 页面播放信息不是有效 JSON

早期解析 `window.__playinfo__` 时，如果只是搜索字符串，可能误命中这类非赋值代码：

```js
if (window.__playinfo__) {
  primarySetting.prefetch = { playUrl: window.__playinfo__ }
}
```

这会把后续代码当成 JSON 解析，导致“页面播放信息不是有效 JSON”。

修复方式：

- 只匹配真正的 `window.__playinfo__ = {...}` 赋值。
- 用括号深度扫描提取完整 JSON 对象。
- 页面没有赋值时，不报 JSON 错，而是走公开播放地址接口兜底。

### 6.2 解析成功但下载返回 400

下载失败的主要原因是页面播放信息提取不稳定，或只拿到了不适合直接下载的分离流。

修复方式：

- 下载前重新解析媒体信息。
- 缺少页面 `__playinfo__` 时走 `x/player/playurl`。
- DASH 分离流下载后用 `ffmpeg` 合并。
- 普通 MP4 `durl` 流标记为 `audio_included=True`，直接下载，不再错误要求额外音频流。

### 6.3 原视频下载很糊

早期 `source` 可能只选到匿名 DASH 中的低清视频流。对示例视频，匿名 DASH 只返回低清，但普通 MP4 接口能返回 720P。

修复方式：

- 同时拉取普通 MP4 `durl`。
- 将普通 MP4 合并到候选视频流。
- `source` 选择当前可用候选里最高清晰度的 MP4。
- 将前端/后端文案从“原视频 MP4”调整为“最高可用 MP4”，避免承诺不存在的高画质。

### 6.4 封面显示失败

封面 URL 已正确解析，例如 `https://i0.hdslb.com/...jpg`。但浏览器从本地前端加载图片时会带 `Referer: http://127.0.0.1:5173/`，B 站图片 CDN 会因 Referer ACL 返回 403，导致页面显示破图。

修复方式：

- 前端封面 `<img>` 增加 `referrerPolicy="no-referrer"`。
- 这样浏览器请求封面时不带本地来源，B 站图片 CDN 可正常返回图片。

如果未来遇到更严格的图片防盗链，再考虑增加受控封面代理，但代理必须限制目标域名，避免引入 SSRF 风险。

## 7. 验证记录

已做过的关键验证：

- `https://www.bilibili.com/video/BV1mAAmzqEfP/` 可解析出标题、作者、时长和封面。
- 示例视频解析后 `source`、`720p`、`audio` 可用，`1080p` 和 `4k` 不可用。
- 通过前端代理下载 `720p`，返回 `video/mp4`，文件约 15 MB。
- `ffprobe` 验证下载文件为 H.264，分辨率 1280x720。
- 封面图带 `localhost` Referer 会 403，不带 Referer 会 200。
- 后端测试：`.venv/bin/python -m pytest backend/tests`，13 个测试通过。
- 前端构建：`npm run build` 通过。

## 8. 后续建议

- 给 `QualityOption` 补充真实高度、码率或文件大小，让前端显示“最高可用 720P MP4”这类更准确文案。
- 将 Provider 错误升级为结构化错误码，方便前端区分登录限制、风控、清晰度不可用和网络失败。
- 如果要引入第三方解析 API，应作为单独 Provider，并默认关闭。
- 如果要支持登录态或高画质，先讨论安全边界，不要新增普通用户 Cookie 输入框。
